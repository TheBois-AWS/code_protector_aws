WebSocket Appendix

Realtime channels are backed by API Gateway WebSocket and Lambda. Clients query /api/ws/config to discover endpoint and then connect with route/channel parameters.

Connection Pattern

# 1) Discover endpoint
GET /api/ws/config

# 2) Connect with query
wss://<endpoint>?path=/api/ws/user/<userId>&token=v2....
# or
wss://<endpoint>?channel=workspace&workspaceId=<workspaceId>&token=v2....

Supported Targets

Target Query Inputs Access Rule
User channel path=/api/ws/user/:userId or channel=user&userId=... Token user must match userId
Workspace channel path=/api/ws/logs/:workspaceId or channel=workspace&workspaceId=... User must have workspace access with view_logs permission

Event Shapes

{
  "type": "WORKSPACE_UPDATE",
  "data": {
    "workspaceId": "...",
    "source": "project"
  }
}

{
  "type": "PROFILE_UPDATE",
  "data": {
    "userId": "..."
  }
}

Reconnect Guidance

  • Use exponential backoff with upper bound (dashboard/workspace clients already do this).
  • Treat websocket as optimization. Always support data refresh via HTTP fallback.
  • On auth failure, clear token and redirect to login.
